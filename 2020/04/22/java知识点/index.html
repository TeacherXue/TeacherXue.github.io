#Java面向对象知识点

一、Javabeen 的要求 ：

  1. 类是公共的
  2. 有一个无参的公共的构造器
  3. 有属性，且有对应的get、set方法

###二、重写的规定

  1. 子类重写的方法的方法名和形参列表与父类被重写的方法的方法名和形参列表相同
  2. 子类重写的方法的权限不小于父类被重写的方法的权限修饰符（但子类不能重写父类的private的方法）
  3. 返回值类型：
  • 父类的返回值类型是void 子类重写的返回值类型也只能是void。
  • 父类被重写的返回值类型是A类型 ，则子类重写的返回值类型可以是A类和其子类。
  • 如果父类返回值类型是基本数据类型，则子类也必须是相同的返回值类型。
  1. 子类重写的方法类型不大于被重写父类的方法抛出的异常类型
  2. 子类和父类中同名同参的方法要么都声明为非static的（考虑重写），要么都声明为非static的（不是重写），static的修饰的不能重写。
  3. 只有方法可以重写，属性是不能重写的！
  4. 重写和重载除了概念上的不同，最重要的是重写能表现出多态性。

###三、四种访问权限

###四、子类对象实例化的过程

1.从结果上来看：

子类集成父类以后，就获得了父类的属性和方法，创建子类对象，在堆空间中，就会加载父类声 明的属性

2.从过程上来看：

当通过子类的构造器创建对象时，一定会直接或者间接的调用其父类的构造器，进而调用父类的构造器，直到调用了java.lang.object类中空参的构造器，所以才可以看到内存中父类的结构，子类对象才可以调用。

###五、多态性

1.何为多态性：

对象的多态性，父类的引用指向子类的对象。 多态性在代码中主要是为了代码的通用性。例如Object类的equals方法

2.多态性的使用，虚拟方法调用：

有了对象的多态性以后，在编译期，只能调用父类中声明的方法，但在运行期，我们实际执行的是子类重写父类的方法。

3. 总结：

编译看左边，运行看右边。（不适用于属性，调用属性的时候都看左边，也就是父类的属性）

###六、向下转型（instanceof 关键字使用）

a instanceof A : 判断对象a是否是类A的实例，如果是，返回true。
使用场景：为了避免在向下转型时候出现ClassCastException的异常，我们在向下转型值钱，先进行instanceof的判断。

###七、Object类

  1. Object类是java中所有类的根父类
  2. 如果在类得声明中未使用extends ，则默认父类是object
  3. Object类中的功能（属性、方法）具有通用型
    属性：无
    方法：equals() / toString() / getClass() / hashCode() / clone / finalize() / wait()、 notify() 、 notifyAll()
  4. Object类中只声明了一个空参构造器

###八、== 和 equals() 的区别 （关联类比toString）

== 用来比较基本数据类型时是判断两边值是否相等 ， 用来比较对象时是判断两个对象地址是否相等
equals() 用来对比两个对象也是看两个对象地址值是否相等，但像Stirng，Date，File， 包装类等都重写了Object类中的equals()方法。重写以后比较的不是两个地址引用是否相等，而是比较两个对象的“实体内容”是否相等

###十一、何时使用static修饰

属性：当属性是可以被多个对象所共享的，不会随着对象的不同而不同
方法：1. 操作静态属性的方法，通畅设置为static 2. 工具类中的方法，习惯上声明为static

###十二、单例的使用场景

网站计数器 ， 应用程序日志， 数据库连接池， windows的任务管理器和回收站

###十三、 代码块

  1. 静态代码块：static{} 随着类的创建而加载 ， 而且只执行一次。 作用：初始化类的信息
  2. 如果定义了多个代码块 ，按定义顺序执行
  3. 动态代码块：{} 随着对象的创建而执行 ，每次创建对象都会执行。作用是给对象属性进行初始化
  4. 属性可以赋值的位置：1. 默认初始化 2. 显示初始化 3.构造器中初始化 4.构造对象后 点语法赋值 5 .在代码块中赋值
  5. 静态代码块内只能调用静态方法和属性 ， 动态代码块内所有的都能调用

###十四、final关键字

  1. Final 可以用来修饰结构：类、方法、变量
  2. Final 用来修饰类 ： 此类不能被其他的类继承 ， 例如String 、 System 、 StringBuffer 类
  3. final 用来修饰方法： 表明方法不可以被重写
  4. Final 用来修饰变量： 此时”变量” 就成为一个常量
    4.1 final修饰属性 ：可以考虑赋值的位置有：显示初始化、代码块中初始化、构造器中初始化
    4.2 final修饰局部变量：尤其是使用final修饰形参时候，表明此形参是一个常量。当我们调用次方法时，给常量形参赋一个实参。一旦赋值以后，就只能在方法体内使用此形参，而不能重新赋值。

###十五、abstract 抽象

  1. 可以用来修饰类和方法
  2. abstract 修饰类： 1. 此类不能实例化 2.抽象类中一定有构造器,便于子类实例化 3. 开发中,都会提供抽象类的子类,让子类对象实例化,完成相关操作.
  3. abstract修饰方法: 只有方法的声明,但没有方法提 包含抽象方法的类,一定是抽象类. 反之抽象类中可以没有抽象方法
  4. 若子类重写了父类所有的抽象方法, 此子类就可以实例化 , 若没有重写所有的抽象方法,则此类也还是一个抽象类,需要使用abstract修饰.

###十六、abstract注意点

  1. Abstract 不能用来修饰属性和 构造器
  2. abstract 不能用来修饰私有方法和静态方法和 final的方法 final的类

###十七、接口 interface

  1. Java中,接口和类是两个并列的结构
  2. 如何定义接口,定义接口中的成员
  3. JDK7及以前:只能定义全局常量和抽象方法

全局常量: public static final , 但是书写的时候可以省略不写

抽象方法: public abstract的

  1. JDK8: 除了定义了全局常量和抽象方法之外 ,还可以定义静态方法 和 默认方法

(1) 接口中定义的静态方法,只能通过接口调用 public static void xxx() {};

(2) 通过实现类的对象,可以调用接口中的默认方法, 并且默认方法是可以被实现类重写的 public default void xxx() {};

(3) 如果子类(或实现类)继承的父类和实现的接口中声明了同名同参数的方法, 那么子类在没有重写此方法的情况下,默认调用的是父类的方法. (如果是属性,则会报错 )

(4) 如果实现类同时实现了多个接口,并且这多个接口中有同名同参数的方法, 如果此时实现类不重写这个方法,则会报”接口冲突”的错误,此时需要重写次方法.

(5) 接4: 如果想调用接口中的同名默认方法,使用 XxxInterface.super.func();

  1. 接口中不能定义构造器! 意味着接口不可以实例化
  2. Java 中 , 接口通过让类去实现(implements)的方式使用.

如果实现类覆盖了接口中的所有抽象方法,则实现类就可以实例化

如果实现类没有覆盖所有的抽象方法, 则此类仍为一个抽象类

  1. Java类可以实现多个接口, 弥补了java单继承的局限性
  2. 接口与接口之间可以继承,并且是可以多继承

      1. 像数据库连接, 输入输出流 , 网络编程中的socket , JVM是不能自动回收的 ,我们需要手动资源释放. 此时 资源释放最好放到finally中.

      3. 一旦try中的异常对象匹配到某一个catch时, 就进入catch中进行异常的处理. 一旦完成处理, 就跳出当前的try catch 结构(在没有finally的情况下), 执行后续代码

        catch中的异常类型如果满足子父类关系,则子类一定要声明在父类的上面,否则会报错
      5. 常用的异常处理方式 :
        ① String getMessage()
        ② e. printStackTrace()
      6. 体会1: 使用try-catch-finally 处理编译时异常, 使得程序在编译时就不再报错,但是运行时仍可能报错, 相当于我们使用try-catch-finally 将一个编译时可能出现的异常,延迟到运行时出现.
      7. 体会2: 开发中, 由于运动时异常比较常见, 所以我们通常就不针对运行时异常写try-catch ,针对编译时异常, 一定要考虑异常处理.

###二十一、throws异常

  1. “throws + 异常类型” 写在方法的声明处, 指名此方法执行时, 可能会抛出异常类型.
    一旦当方法体执行时, 出现异常, 仍会在异常代码处生成一个异常类得对象, 此对象满足throws后异常类型时,就会抛出. 异常代码后续的代码不会再执行.
  2. 体会: try-catch-finally :真正的将异常处理掉了,
    throws 方法只是把异常抛给了方法的调用者, 并没有处理.
  3. 开发中如何选择使用try 还是 throws
    1. 如果父类中被重写得方法没有throws方式处理异常, 则子类重写的方法也不能用throws , 意味着如果子类重写的方法有异常, 必须使用try方式处理.
    2. 执行方法a时, 先后又调用了另外的几个方法, 这几个方法是递进关系执行的. 建议这几个方法使用throws的方式进行处理.
      而执行方法a 可以考虑使用try - catch 方式处理.

思考: synchronized与lock的异同

​ 相同: 二者都可以解决线程安全问题

​ 不同: synchronized机制在执行完相应的代码后, 自动的释放同步监视器

#####同步方法 : synchronized修饰静态方法和普通方法的区别

synchronized具有同步功能，是一种互斥锁，锁的是对象，synchronized修饰普通方法时，锁对象是this对象。
修饰静态方法时，锁对象是字节码文件对象。

synchronized可以保证原子性，有序性，可见性。

#####synchronized的缺点：

  1. synchronized底层是由jvm实现，因此不能手动控制锁的释放，不如lock锁灵活，synchronized修饰的方法一旦出现异常，jvm保证锁会被释放（lock锁需要在finally中释放）。
  2. synchronized是非公平锁，不保证公平性。

七、线程死锁

  1. 不同的线程分别占用对方需要的同步资源不放弃,都在等待对方放弃自己需要的同步资源锁, 就形成了线程死锁
  2. 出现死锁后,不会出现异常,不会出现提示, 只是所有的线程都处于阻塞状态, 无法继续

#####解决方法

  1. 专门的算法 或者 原则
  2. 尽量减少同步资源的定义
  3. 尽量避免嵌套同步

八、线程间通信

  1. #####涉及到的三个方法

    1. wait(); 一旦执行此方法, 当前线程就进入阻塞状态, 并且释放同步监视器
    2. notify(); 一旦执行此方法, 就会唤醒被wait()的一个进程, 如果有多个进程被wait , 就会唤醒优先级高的那一个.
    3. notifyAll(); 此方法执行后会唤醒所有被wait的线程.

  2. #####说明

    1. 三个通信方法 只能出现在同步代码块或者同步方法中 (lock和unlock之间都不行)
    2. 三个方法的调用者必须是同步代码块或者同步方法中的同步监视器
    3. 三个方法都是定义在Object类中

  3. #####sleep(); 和 wait(); 方法的异同

    1. 相同点: 一旦执行方法, 都可以阻塞当前线程
    2. 不同点:
      1. 两个方法声明的位置不同. Thread类中声明sleep() , Object类中声明的wait()
      2. 调用范围不同: sleep 任何需要的地方调用, wait()只能在同步代码块或者同步方法中
      3. sleep必须捕获异常，而wait，notify和notifyAll不需要捕获异常
      4. 关于是否释放监视器: 如果两个方法都使用在同步代码块或者方法中, sleep是不会释放同步监视器, 而wait执行后会释放.

一. String

  1. String声明为final表明为不可继承的
  2. String实现了Serializable接口: 表示字符串是支持序列化的
    实现了Comparable接口: 标识String可以比较大小
  3. String内部定义了final char[] value 用于存储字符串数据
  4. String: 代表不可变的字符序列. 简称: 不可变性 (字符串再内存方法区中的字符串常量池里面)
    1. 当对字符串进行重新赋值时, 需要重写指定内存区域赋值, 不能使用原有的value进行赋值 (字符串对象地址改变)
    2. 当对现有的字符串进行连接操作时,也需要重新指定内存区域赋值, 不能使用原有value赋值
    3. 调用replace方法时候, 修改字符或字符串时 , 不能使用原有value赋值.
  5. 通过字面量(区别于new)的方式给字符串赋值, 此时字符串值声明在字符串常量池中
  6. String的实例化方式:
    1. 通过字面量定义 String s = “abc”; (方法区的字符串常量池中)
    2. 通过new + 构造器方式 ; (堆区)
  7. 字符串的特性

二. StringBuffer&StringBuilder&String

Stirng: 不可变得字符序列, 底层使用char[] 存储

StringBuffer: 可变, 线程安全, 效率低, 底层使用char[] 存储

StringBuilder: 可变, jdk5.0新增 , 线程不安全, 但是效率高, 底层使用char[] 存储

  1. 源码分析:
    1. String str = new String();// char[] value = new char[0];
      String str1 = new String(“abc”); // char[] value = new char[]{‘a’, ‘b’, ‘c’};
    2. StringBuffer sb1 = new StringBuffer(); // char[] value = new char[16]; //底层创建了一个长度是16的char类型数组
      StringBuffer sb2 = new StringBuffer(“abc”); // char[] value = new char[“abc”.lenght() + 16];
    3. 问题1 : Sys.out.println(sb2.length()); // 3 + 16
      问题2 : 扩容问题: 如果要添加的数据底层数组成不下 , 就需要扩容底层数组, 默认情况下 扩容为容量的2倍+2 , 同时数组中的元素复制到新数组中.
    4. 开发中建议使用 : StringBuffer(int capatity) 来构建 , 提前预估大小可以提升性能

五. Comparable 和 Comparator

  1. Comparable接口的使用举例:
    1. 像String , 包装类等实现了Comparable接口, 重写了compareTo()方法, 给出比较两个对象的方法
    2. 像String ,包装类等重写compareTo()方法以后, 进行了从小到大的排列
    3. 重写compareTo(obj) 的规则:
      1. 如果当前this对象大于obj, 返回整数
      2. 如果当前this对象小于obj, 返回负数
      3. 相等, 返回0
  2. Comparator接口的使用: 定制排序
    1. 当Comparable实现不了当前的排序需求时, 那么就需要用comparaor的对象来排序
    2. 重写compare(Object o1 , Object o2) 方法, 比较o1 和 o2 的大小
      1. 如果方法返回整数,则 o1 > o2
      2. 如果返回0 , 表示相等
      3. 返回负数 , 表示o1 小于 o2
